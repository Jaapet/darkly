Context:
XSS happens when the browser interprets user-controlled content.
On /index.php?page=media&src=nsa, the src parameter is used as a media source, so I tested whether it could be made to load something other than an intended resource.

Method:
1. Test a Data URI (embedded content)
    A Data URI directly embeds content (a common test when you control src):
    data:text/plain,hello and data:text/html,<h1>hello</h1>
    Result: “hello” is displayed → 'data:' scheme is accepted, 'text/html' is interpreted by the browser.
    HTML injection is possible.

2. Inject script:
    Try same with <script>alert(1)</script>
    We have got a flag message appearing but we can not see the flag.
    Result: Javascript is executed, filter is insufficient, XSS is confirmed.

3.Switch to base64
    Raw HTML in a URL can break (<, >, etc.). Base64 is more stable.
    echo -n '<script>alert(1)</script>' | base64
    PHNjcmlwdD5hbGVydCgxKTwvc2NyaXB0Pg== then injected it via src=data:text/html;base64,PHNjcmlwdD5hbGVydCgxKTwvc2NyaXB0Pg==
    Result: flag obtained.
    

Fix
-Validate src using a strict allowlist.
-Do not let users directly control a browser-loaded source.
-Reinforce with a Content Security Policy (CSP).
-Map user input to predefined server-side resources
